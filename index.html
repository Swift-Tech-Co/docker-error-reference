Error Category Cause Fix
Generic Application Error Exit Code The application inside the container exited with a non-zero status. Usually an unhandled exception, missing config, or startup failure. Run `docker logs ` to see the application's stdout/stderr. Check entrypoint script, env vars, and application startup sequence.
OOM Kill / SIGKILL Exit Code Process received SIGKILL. Either the container exceeded its memory limit and was killed by the OOM killer, or the container was manually stopped with `docker kill`. Increase memory limit with `--memory` flag. Profile the application for memory leaks. Check `docker inspect ` for OOMKilled: true.
Segmentation Fault (SIGSEGV) Exit Code The process attempted to access memory it was not permitted to access. Common in C/C++ applications, corrupted binaries, or incompatible base images. Check for architecture mismatch (arm64 vs amd64). Rebuild the image for the target platform. Run with `--platform linux/amd64` if needed.
Graceful Shutdown (SIGTERM) Exit Code Container received SIGTERM and exited cleanly. This is the expected behaviour when `docker stop` is called. No action needed if this is expected. If unexpected, check orchestration system (Kubernetes, ECS) for evictions or rolling restarts.
Command Not Found Exit Code The command specified in CMD or ENTRYPOINT was not found in the container. Missing binary or incorrect path. Verify the entrypoint binary exists in the image: `docker run --rm which `. Check spelling and PATH in the Dockerfile.
Permission Denied on Entrypoint Exit Code The entrypoint script or binary exists but is not executable. Add `RUN chmod +x /entrypoint.sh` in the Dockerfile before the ENTRYPOINT instruction.
Misuse of Shell Builtin Exit Code Shell command misuse or incorrect usage of a shell builtin. Also returned by some applications to indicate a general usage error. Check your CMD or ENTRYPOINT for syntax errors. Inspect logs with `docker logs `.
Image Pull Failed Image Image name is misspelled, tag does not exist, or registry authentication failed. Verify the image name and tag with `docker pull :`. Login to private registries with `docker login `. Check network connectivity to the registry.
Image Manifest Unknown Image The specified tag does not exist in the registry, or the image was deleted after being referenced. Check available tags on Docker Hub or your registry. Use `docker images` to check local cache. Use `latest` or a specific valid tag.
Image Platform Mismatch Image Pulling an image built for a different CPU architecture (e.g. ARM image on AMD64 host or vice versa). Build multi-platform images with `docker buildx`. Pull with `--platform linux/amd64` flag. Use `DOCKER_DEFAULT_PLATFORM=linux/amd64` env var.
Port Already Allocated Network Another process or container is already bound to the same host port. Run `lsof -i :3000` or `ss -tulpn | grep 3000` to find the conflicting process. Stop the other container or use a different host port: `-p 3001:3000`.
Network Not Found Network The specified Docker network does not exist, usually because `docker network create` was not run or the network was deleted. Create the network: `docker network create `. In Compose files, declare the network in the `networks:` section.
DNS Resolution Failure Network Container cannot resolve DNS names. Common when containers on different networks try to communicate by service name. Ensure all communicating containers are on the same Docker network. Use `--network` flag or define shared networks in docker-compose.yml. Check `/etc/resolv.conf` inside the container.
No Space Left on Device Storage Docker data directory has filled the host disk. Common after many builds accumulate unused layers and stopped containers. Run `docker system prune -af` to remove unused images, containers, and build cache. Check disk usage with `df -h` and `docker system df`.
Volume Mount Failed Storage The host path specified in a bind mount does not exist, or Docker Desktop does not have permission to access it. Create the host directory before running the container. On Docker Desktop (Mac/Windows), ensure the path is in the File Sharing list under Preferences.
Build Context Too Large Build The build context sent to the daemon includes large files (node_modules, .git, datasets) that should be excluded. Add a `.dockerignore` file in the build directory. Exclude `node_modules`, `.git`, `*.log`, build artifacts, and large data files.
RUN Command Failed During Build Build A RUN instruction in the Dockerfile failed. Common causes: package not found, wrong base image version, network issues during apt/pip install. Run the failing command interactively: `docker run --rm -it bash`. Check package names and versions. Add `--no-cache` to the build to skip stale layers.
Unexpected Cache Invalidation Build COPY or ADD instructions placed before RUN apt/pip install force reinstallation on every code change. Order Dockerfile instructions from least-changing to most-changing. Copy dependency manifests (package.json, requirements.txt) first, install dependencies, then copy application code.
Docker Daemon Not Running Daemon The Docker daemon process is not started, or the current user does not have permission to access the socket. Start Docker: `sudo systemctl start docker`. Add user to docker group: `sudo usermod -aG docker $USER`. On Desktop, open the Docker application.
Docker Daemon Out of Memory Daemon The Docker host does not have enough free memory to start the container. Stop unused containers: `docker stop $(docker ps -q)`. Increase host RAM. Set container memory limits to avoid any single container consuming all available memory.
Container Name Already in Use Runtime A stopped container with the same name still exists. Remove the existing container: `docker rm myapp`. Or use `--rm` flag so the container auto-removes on exit. Or use `docker rename` to keep the old one.
Container Health Check Failing Runtime The HEALTHCHECK instruction in the Dockerfile is failing. The application may not be ready or crashed silently. Inspect health logs: `docker inspect --format='{{json .State.Health}}' `. Adjust `--health-start-period` to give the app more startup time. Debug the healthcheck command manually.
Compose File Version Unsupported Compose The Compose file specifies a version that the installed Docker Compose version does not support. Update Docker Compose to the latest version. Or lower the `version:` field in docker-compose.yml. Compose v2 (bundled with Docker) uses the Compose Specification and ignores the version field.
Compose Service Failed to Start Compose One of the services in a docker-compose.yml failed during build or startup. Run `docker compose logs ` to isolate the failing service. Start services individually with `docker compose up ` to narrow down the issue.
Service Started Before Dependency Ready Compose `depends_on` only waits for the container to start, not for the service inside to be ready. A database container starting does not mean Postgres is accepting connections yet. Add a `healthcheck` to the dependency service and use `depends_on: condition: service_healthy` in Compose v3.9+. Or add a startup retry loop in the dependent application.
Docker Hub Rate Limit Registry Anonymous Docker Hub pulls are rate-limited to 100 pulls per 6 hours per IP. CI/CD environments sharing an IP can hit this quickly. Login to Docker Hub in CI: `docker login -u $DOCKER_USER -p $DOCKER_TOKEN`. Use a Docker Hub Pro account. Mirror images to a private registry. Use `--cache-from` to reduce pull frequency.
TLS Certificate Error Registry Attempting to connect to a private registry using a self-signed certificate that Docker does not trust. Add the CA cert to `/etc/docker/certs.d//ca.crt`. Or add the registry to `insecure-registries` in `/etc/docker/daemon.json` (not recommended for production).
Exec Format Error Runtime The binary in the container was compiled for a different architecture than the host. Common when building on Apple Silicon (arm64) and deploying to x86_64 servers. Build for the target platform: `docker buildx build --platform linux/amd64 -t myimage .`. Use multi-stage builds to compile inside the container rather than copying host binaries.
Operation Not Permitted (Capabilities) Security The container is attempting a privileged operation (e.g. binding port < 1024, modifying kernel parameters) that requires Linux capabilities not granted by default. Grant specific capabilities with `--cap-add NET_BIND_SERVICE` rather than running `--privileged`. For sysctl changes use `--sysctl net.ipv4.ip_forward=1`.
Syscall Blocked by Seccomp Security Docker's default seccomp profile blocks certain syscalls. Some applications (perf, certain JVM operations, ptrace) require syscalls not in the default allowlist. Run with a custom seccomp profile: `--security-opt seccomp=custom.json`. As a last resort: `--security-opt seccomp=unconfined` (not recommended for production).
cgroup Resource Limit Exceeded Runtime Container exceeded CPU, memory, or PID limits set by cgroup constraints (via Docker `--memory`, `--cpus`, `--pids-limit` flags or Kubernetes resource limits). Increase limits: `--memory=2g --cpus=1`. Profile the application for resource leaks. Use `docker stats ` to monitor live resource usage.
COPY / ADD Source Not Found Build The file specified in a COPY instruction does not exist in the build context, or it was excluded by .dockerignore. Check the file path relative to the build context directory. Review .dockerignore for accidental exclusions. Verify the file exists locally before building.
Image Layer Too Large Build Files added in one RUN layer are deleted in a subsequent layer, but the data still exists in the earlier layer (layers are additive). Combine RUN commands with `&&` and clean up in the same layer: `RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*`. Use multi-stage builds.
Log Driver Configuration Error Daemon The configured log driver (fluentd, awslogs, gelf) is unreachable or misconfigured. Verify the log driver endpoint is reachable from the Docker host. Check log driver options in daemon.json or container `--log-driver` and `--log-opt` flags.
OverlayFS Error Storage OverlayFS storage driver error, often caused by filesystem incompatibility (e.g. XFS without d_type support, ZFS, NFS). Check filesystem type: `df -T /var/lib/docker`. Ensure XFS is formatted with `ftype=1`. Consider switching to a supported storage driver for your filesystem.
Swarm Not Initialized Swarm Attempting to run Swarm commands on a node that has not been initialized as a Swarm manager. Initialize Swarm: `docker swarm init`. For worker nodes, use the join token from `docker swarm join-token worker`.
Network Subnet Conflict Network The subnet Docker is trying to allocate for a new network overlaps with an existing network or host route. Specify a custom subnet: `docker network create --subnet 172.28.0.0/16 mynet`. Or prune unused networks: `docker network prune`.
Health Check Timeout Runtime The healthcheck command took longer than the configured `--health-timeout` value (default 30s). Increase timeout: `HEALTHCHECK --interval=30s --timeout=60s CMD ...`. Optimize the health check command. Ensure the application is not under excessive load.
Required Environment Variable Missing Runtime The application requires an environment variable that was not passed to the container. Pass env vars with `-e VAR=value` or `--env-file .env`. In Compose, use the `environment:` or `env_file:` keys. Validate required vars in entrypoint scripts with `set -u`.
File Permission Denied Inside Container Runtime The process inside the container is running as a non-root user without write permission to the target directory. Add `RUN chown -R appuser:appuser /app` before switching to the non-root user in your Dockerfile. Or set directory permissions with `RUN chmod 755 /data`.
Zombie Processes (PID 1 Problem) Runtime When a shell script is PID 1 in a container, it does not reap child processes or forward signals correctly, causing zombie accumulation and unclean shutdowns. Use `exec` in entrypoint scripts: `exec node server.js`. Or use a lightweight init system: `docker run --init`. Or use `tini` as PID 1 in the Dockerfile.
Open File Limit Exceeded (ulimit) Runtime The process inside the container hit the open file descriptor limit. Common with high-traffic servers, databases, or applications that open many connections. Increase limits: `docker run --ulimit nofile=65535:65535`. Set system-wide defaults in `/etc/docker/daemon.json` under `"default-ulimits"`. Check current usage: `ls /proc//fd | wc -l`.
